# v0.10 — verdict slot indexed by `(vk ^ ts)` silently misses across a tick

**One-line:** Mixing the timestamp into the verdict slot index, combined with a tight (<10 ms) freshness window, makes the DLL miss valid verdicts whenever `GetTickCount()` advances by its quantum (~15 ms) between the LL classifier and the per-process hook. Result: every rule with `sourceFilter ≠ ANY` silently becomes a no-op.

## The broken pattern

In `include/mapping_defs.h`:

```c
// WRONG (v0.10 draft 1)
#define KV_VERDICT_RING_SIZE 64
#define KV_VERDICT_RING_MASK 0x3F
#define KV_VERDICT_WINDOW_MS 5
```

In `src/shared_mem.c`:

```c
// WRONG — writer
int slot = (int)((vkCode ^ timestampMs) & KV_VERDICT_RING_MASK);
g_shared->ring[slot] = ...;
```

In `src/hook/hook.c`:

```c
// WRONG — reader
uint32_t now = (uint32_t)GetTickCount();
for (uint32_t back = 0; back <= KV_VERDICT_WINDOW_MS; back++) {
    uint32_t ts = now - back;
    int slot = (int)(((uint32_t)vk ^ ts) & KV_VERDICT_RING_MASK);
    const KVEventVerdict *v = &g_shared->ring[slot];
    if (v->vkCode == (uint32_t)vk &&
        (now - v->timestampMs) <= KV_VERDICT_WINDOW_MS)
    {
        // ...
    }
}
```

Two compounding problems:

1. **Slot collisions across different VKs.** Mask is 6 bits. `F1 (0x70)` and `0 (0x30)` both map to slot `0x30`. A `VK_0` press overwrites an `F1` verdict.
2. **GetTickCount granularity beats the window.** Default tick is ~15.6 ms. The LL classifier writes `timestampMs = kb->time = T`. A few hundred microseconds later the DLL runs and calls `GetTickCount()` → may already return `T + 15`. The probe loop searches `back ∈ [0..5]` for matching slot, i.e. timestamps `[T+10 .. T+15]`. The actual write was at `T`. **No probe finds it.** Even if it did, the freshness check `(now - v->timestampMs) ≤ 5` rejects it (delta is 15).

## Failure scenario

1. User creates rule "RWin+RAlt → F1, sourceFilter = REMOTE".
2. User sends RWin+RAlt over a KVM/RDP session (real `LLKHF_INJECTED`).
3. LL classifier writes slot `(RAlt ^ T) & 0x3F` with `flags = LLKHF_INJECTED`.
4. ~600 µs later, focused process's WH_KEYBOARD receives RAlt-down.
5. The DLL calls `GetTickCount()` → returns `T + 15` because the timer ticked over.
6. The probe loop tries slots derived from `T + 15` down through `T + 10`. None match the slot where the classifier actually wrote (derived from `T`).
7. `kv_lookup_verdict` returns 0 → `isInjected = 0`.
8. The rule's `sourceFilter == KV_SOURCE_REMOTE` check sees `!isInjected` → **skip**.
9. The remote key is treated as if no rule matched. The whole feature is silently dead.

Hard to detect: nothing crashes, no log line, just "sourceFilter feels broken".

## The fix

Drop the timestamp from the slot index. Use one slot per VK code. Lookup is a single read. The freshness check stays for stale-slot rejection, with the window widened to cover at least two tick quanta:

```c
// mapping_defs.h — RIGHT
#define KV_VERDICT_RING_SIZE 256
#define KV_VERDICT_RING_MASK 0xFF
#define KV_VERDICT_WINDOW_MS 30   // ~2 GetTickCount quanta
```

```c
// shared_mem.c — RIGHT
int slot = (int)(vkCode & KV_VERDICT_RING_MASK);
```

```c
// hook.c — RIGHT
const KVEventVerdict *v = &g_shared->ring[(unsigned)vk & KV_VERDICT_RING_MASK];
if (v->vkCode != (uint32_t)vk) return 0;     // fingerprint must match the VK
uint32_t now = (uint32_t)GetTickCount();
if ((now - v->timestampMs) > KV_VERDICT_WINDOW_MS) return 0;  // stale
```

Cost: the table grows from 64 × 16 B = 1 KB to 256 × 16 B = 4 KB. Negligible. The hot path goes from a 6-iteration probe to a single read.

## STOP SIGN

If you're about to:

- Shrink `KV_VERDICT_RING_SIZE` below 256 ("we don't need a slot for every VK")
- Tighten `KV_VERDICT_WINDOW_MS` below ~25 ("5 ms is plenty")
- Re-introduce `vk ^ ts` slot indexing because "it spreads collisions better"
- Call `timeBeginPeriod(1)` to "fix" the tick granularity instead of widening the window — that's a system-wide setting, do not do it

→ **Stop and re-read this note.** This bug is invisible (no crash, sourceFilter just silently no-ops). The 256-slot single-read design exists because every "smarter" indexing was tried and produced flaky verdicts on real hardware.

## Related lesson

Whenever you see `GetTickCount` in concurrency code, the relevant question is "what is my window vs the quantum?" Default quantum is 10–16 ms depending on whether anything (audio drivers, browser scheduler) has raised the timer resolution. Assume 16 ms unless you explicitly know otherwise.
