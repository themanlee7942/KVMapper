# v0.10 — auto-repeat collision can erase our verdict slot

**One-line:** Removing `KV_TLS int g_dispatching` from `src/hook/hook.c` allows infinite loops when a user holds down a remapped key. The verdict-ring lookup is NOT sufficient on its own.

## The broken pattern

When v0.10 was first written, the thread-local `g_dispatching` flag was deleted because "the verdict ring already tells us if an event is ours":

```c
// hook.c — WRONG (v0.10 draft 1, briefly committed and reverted)
LRESULT CALLBACK KeyboardProc(int code, WPARAM wParam, LPARAM lParam) {
    if (code < 0) return CallNextHookEx(...);
    if (!g_shared) return CallNextHookEx(...);

    // No g_dispatching check - rely entirely on the verdict ring.
    BYTE vk = (BYTE)(wParam & 0xFFu);
    int isInjected = 0, isOurs = 0;
    kv_lookup_verdict(vk, &isInjected, &isOurs);
    if (isOurs) return CallNextHookEx(...);  // <-- assumed sufficient
    ...
}
```

And in `src/hook/inject.c`:

```c
// inject.c — WRONG (no TLS flip around SendInput)
void kv_fire_action(const KVMapping *m) {
    ...
    SendInput((UINT)idx, inputs, sizeof(INPUT));  // <-- no g_dispatching guard
}
```

## Failure scenario

1. User has a rule "A → A" (silly but legal; same shape applies to any self-referential mapping).
2. User holds **A**. Windows starts auto-repeating, posting a `WM_KEYDOWN(A)` every ~30 ms.
3. Each user-A flows through the LL classifier, which writes slot `[A]` with `isOurs = 0`.
4. The DLL fires the rule, calls `SendInput(A)` with our session token in `dwExtraInfo`.
5. The LL classifier sees our synthetic A and writes slot `[A]` with `isOurs = 1`.
6. **But before the DLL's WH_KEYBOARD receives our synthetic A**, the user's next auto-repeat lands at the LL classifier, which overwrites slot `[A]` back to `isOurs = 0`.
7. The DLL's WH_KEYBOARD finally fires for our synthetic A. Reads slot `[A]` → `isOurs = 0`. Treats it as user input. Fires the rule again. Loops.

The race is tight (LL-fire → slot-overwrite → WH_KEYBOARD-dispatch), so it doesn't reproduce every time, but holding a remapped key for a few seconds will hit it.

## The fix

Restore the thread-local flag and flip it around `SendInput`. It runs in microseconds and only protects the **same thread** that called `SendInput` — which is exactly the thread where re-entry happens. The verdict ring still does its job for cross-thread and cross-instance cases; `g_dispatching` is the local belt-and-braces.

```c
// hook.c — RIGHT
KV_TLS int g_dispatching = 0;  // not static; inject.c flips it

LRESULT CALLBACK KeyboardProc(...) {
    ...
    if (g_dispatching) return CallNextHookEx(...);  // BEFORE ring lookup
    ...
}
```

```c
// inject.c — RIGHT
if (idx > 0) {
    g_dispatching = 1;
    SendInput((UINT)idx, inputs, sizeof(INPUT));
    g_dispatching = 0;
}
```

## STOP SIGN

If you're about to:

- Delete `g_dispatching` because "the verdict ring covers this"
- Move the `g_dispatching` check after the ring lookup (it must be first)
- Drop the `g_dispatching = 1` flip in `inject.c` because "no rule will self-fire in practice"

→ **Stop and re-read this note.** All three were tried; all three reintroduce the bug. The verdict ring + TLS flag are complementary, not redundant.
